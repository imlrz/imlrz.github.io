RQ1 · 信息多轮释放 · LLMS GET LOST IN MULTI-TURN CONVERSATION
iclr25 best paper · 2025 May

Motivation

现在的 LLM 在单轮、完整指令下表现很好，但一旦用户把需求拆散到多轮对话里逐步补充，模型性能和稳定性会显著下降。

但在现实中，用户很多时候不会一次性把所有信息说清楚，而是在多轮的交互间补充完全。

例如原始单轮任务可能是
写一个 Python 函数，实现某个功能，要求处理空输入、返回特定格式、复杂度 O(n)、不要使用某个库。
现实中 · 逐轮补充

例如第一轮说：

帮我写一个函数。

然后后面逐轮补充：

输入是一个列表。 需要处理空列表。 返回值要是字典。 复杂度最好是 O(n)。 不要用 pandas。
P1 · LLMs Get Lost in Multi-Turn Conversation
P1 · 把性能下降拆成两个维度

论文把性能下降拆成两个维度：

  • Aptitude：近似“最好情况下模型能做到多好”，用 90th percentile 表示。
  • Unreliability：近似“同一个任务不同模拟轨迹之间波动有多大”，用 90th percentile 和 10th percentile 的差距表示。

论文发现，Sharded 场景下模型平均性能显著下降，但主要原因不是 aptitude 大幅下降，而是 unreliability 暴涨。也就是说，模型并不是完全不会做，而是多轮对话中很容易走偏；一旦早期轨迹错了，后面就很难纠正。论文报告：从 Full 到 Sharded，模型 aptitude 平均只下降约 16%，但 unreliability 平均增加 112%，几乎翻倍。

P1 · 为什么模型会 “get lost”？

论文总结了几个机制，比较有启发：

过早给最终答案

模型在信息还不完整时，就急着生成完整解法。比如用户只说“帮我写个函数”，模型就开始假设输入输出格式、边界条件、复杂度要求。问题在于，这些假设一旦写进对话历史，后面会变成“自我污染”的上下文。

过度依赖自己之前的错误答案

当用户后续补充约束时，模型不是从任务状态重新建模，而是在旧答案上修修补补，结果就是越修越乱。论文把这个叫做 answer bloat：多轮最终答案往往比 Full / Concat 长 20%–300%，但更长不等于更好，反而暗示模型不断叠加补丁、保留错误假设。

中间轮信息丢失

论文还观察到一种 loss-in-middle-turns：模型更容易关注开头和最后几轮，而忽略中间补充的信息。这和 long-context 的 lost-in-the-middle 很像，只是这里发生在 multi-turn conversation 里。

回复太啰嗦

模型每一轮都输出很长的内容，会把自己的假设、解释、半成品答案不断塞进上下文，导致后续轮次的有效用户信息被稀释。这点和我们平时使用很一致：模型越早“自信地展开”，后面越难拉回来。

RQ1 · 用户介入 · Modeling Distinct Human Interaction in Web Agents
CMU · Graham Neubig · 2026 May

Motivation

不要只把 Web Agent 当成全自动执行器，而要显式建模“用户什么时候会介入、为什么介入、介入后是否还会把控制权交还给 agent”。

传统 Web Agent 大多默认一个范式：用户给任务 → agent 自动规划 → agent 自动执行 → 最后返回结果。但真实使用中，用户经常会中途介入，例如：

  • agent 点错了；
  • agent 忽略了用户隐含偏好；
  • agent 面对复杂网页组件卡住了；
  • 用户突然发现自己没有把需求说清楚；
  • 当前操作风险太高，用户想自己接管。

论文认为，现有 agent 的问题不是单纯“不够自动化”，而是 不知道什么时候该自动、什么时候该停下来让用户介入。如果 agent 太自主，它会越过关键决策点；如果 agent 太频繁确认，又会打断用户，降低效率。

P2 · 任务定义 & 数据集

任务定义

预测每一个时间步用户是否会中途介入。

数据集

作者收集了一个真实人类和 Web Agent 协作的数据集 CowCorpus，包含 400 条真实用户 Web 导航轨迹，总计 4200+ 个交错的人类/agent action。

数据采集方式是：20 个用户，每人完成 20 个任务，其中 10 个是从 Mind2Web 选出来的标准任务，10 个是用户自己选择的 free-form tasks。总计就是 20 × 20 = 400 条 human-agent collaborative web trajectories。论文记录每一步是谁做的，是 agent 还是 human，以及时间戳和网页快照。

P2 · 用户介入原因（论文总结成三类）
Error Correction and Recovery

用户介入是为了纠错。比如 agent 点错元素、提前执行不该执行的操作，或者陷入重复无效动作。论文举例说，agent 在产品搜索还没完成前，就过早选择了 location filter；或者 agent 一直循环做无效动作，用户只好接管几步把任务拉回正轨。

Preference Misalignment

用户介入是为了修正偏好不匹配。很多时候用户一开始没有把偏好写完整，比如价格、地点、品牌、筛选条件。agent 执行到中途，用户看到它的行为不符合自己真实偏好，才补充说明。论文举了“买牙膏时用户没提前指定品牌，看到 agent 中间动作后才补充偏好”的例子。

Assistive Intervention

用户介入不是因为 agent 逻辑错了，而是因为网页环境太复杂，比如 dropdown、captcha、dynamic layout、复杂 DOM，或者页面资源没加载出来。用户也可能为了避免不可恢复错误而主动接管。

P2 · 用户协作风格

对用户进行聚类，得到四类协作模式：

类型特征我的理解
Hands-off很少介入，让 agent 基本端到端执行信任 agent，低监督
Hands-on频繁介入，但会和 agent 来回交替控制高监督，共同执行
Collaborative选择性介入，介入后通常交还控制权理想协作型用户
Takeover介入较晚，但一旦介入就不太交还控制权agent 失控后用户接管

RQ1 · 用户不合作 · RealUserSim
Salesforce AI Research · 2026 Apr

Motivation：用户不合作

现在很多 agent benchmark，例如 τ-bench，会用 LLM 扮演用户，让 agent 和这个 simulated user 多轮交互。问题是：这个 simulated user 往往不像真实人类。

论文指出现有 user simulation 有两个核心问题。 →

P3 · 现有 user simulation 的两个核心问题

第一个：Formalism Ceiling

没有约束的 LLM simulator 默认会变成“礼貌、完整、语法正确、合作度很高”的用户。换句话说，它不像真实用户，更像另一个 assistant。论文发现，在没有真实 profile 的 baseline 下，style-related dimensions 的匹配率只有 6%–8%，因为 LLM 会把所有用户都“正规化”为工整表达。

第二个：Directive Amplification

如果 benchmark 手写行为指令，比如“你是一个非常愤怒的顾客”，不同 simulator model 会把这个指令放大成很夸张的表演：有的会写 sobbing、choking back tears 这种舞台动作，有的只是说“我很 upset”。同一个 directive 在不同 simulator 上产生不同强度的行为，导致 benchmark 难度不稳定、不同 simulator 结果不可比。

P3 · 方法

RealUserSim 的方法很直接：从真实人类-LLM 对话里抽取用户行为 profile，然后把这个 profile 注入到 user simulator 的 prompt 里。

它用了 WildChat-4.8M 作为真实对话来源。作者从 3.2M conversations 中筛选出 21,637 条英文、多轮、GPT-4o 轨迹，覆盖 7,311 个 unique users，最后构造了 7,275 个 executable behavioral profiles。

部分内容作用
Demographics年龄、教育、性别、职业、地点等，部分来自显式披露，部分来自推断提供用户背景
Additional background从历史对话中总结的额外背景增强上下文
Communication style instructionsCommand + Example 的风格手册约束语言风格和交互方式

每个 profile 不是一句泛泛的 persona，比如“informal user”，而是一个 Executable Persona Manual。格式是：

Command: 经常重复词语来强调
Examples: "the most the most famous food";
          "there are there are huge differences"

Command: 用 hmm / ahh 表示犹豫
Examples: "hmm i think the older people";
          "hmmm if we eat veegtables"

Command: 不满意时要求更多例子或选项
Examples: "share a more short caption.";
          "share 10 more"

论文分析了 NP 成功但 Real Persona 失败的轨迹，归纳出三类失败机制：

RQ2

如何 Bridge the Gap

RQ2 · FOUND IN CONVERSATION: LLMs Teach Themselves to Close the Multi-Turn Gap

Motivation

既然模型在 single-turn 下会做，为什么 multi-turn 下不会做？能不能把 single-turn 能力迁移到 multi-turn 行为里？

作者的判断是：这个 gap 很大程度上是 behavioral gap，不是 capability gap。因为同一个模型面对同样信息，只是呈现形式不同，single-turn 能做而 multi-turn 不能做，说明能力已经在模型里，只是在多轮交互中没有稳定调用出来。

RQ3 · 我们的目标
  1. 研究为何有些 benchmark 上的高分模型实际表现并不行——除去测试集泄漏，是否有 bench 本身的问题
  2. benchmark 如何改造以指导方法研究 （问题：）
    1. interactive 是否一定会造成性能下降——不是
      1. interactive bench 实际改造过程中，多轮交互的题目反而性能提升
      2. 性能下降的核心不是 interactive，而是用户的不合作（例如 lost in the middle 的信息给予不全）；在 interactive bench 的改造中，有的改造太“合作”了，甚至可以说是在“指导”agent
    2. 是否要 100% 模拟现在用户的行为？
      1. Neubig 的文章研究了“用户接管”这一行为，这一行为是否有必要体现在 eval 中？
——个人人为不用，Human Agent Interactive 的理想状态应该是 vibe coding。除了权限与偏好澄清，agent 不应当主动要求与人类交互。Agent 也没有理由认为人类一定比自己强从而请求人类的指导。c. 可能的切入点：multi-tasking / multi-thread？
